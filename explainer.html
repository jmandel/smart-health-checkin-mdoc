SMART Health Check-in over W3C Digital Credentials API

A request asks for check-in work. A response returns what the patient shared.

Two application roles

The SMART Health Check-in model is deliberately small. A requester describes the check-in data it would like. A responder decides what is available, shows the user a consent experience, and returns the artifacts the user chose to share.

Requester

Usually a clinic, registration kiosk, patient portal, or other relying-party web app.

Owns

purpose item list accepted media types encryption key

It creates a SMART request and invokes the W3C Digital Credentials API.

↔

Responder

Usually a patient wallet or health app that can present clinical data on behalf of the user.

Owns

available data user consent shared artifacts per-item status

It returns a SMART response through the same Digital Credentials flow.

The conversation shape

Think of the exchange as a checklist rather than a document request. The requester sends a list of named items. The responder sends back clinical artifacts and a status for every item on the checklist.

1. Requester chooses purpose Example: clinic check-in, billing, visit intake, prior auth.
2. Requester lists items Patient demographics, insurance card, IPS summary, intake form.
3. Responder matches items The wallet decides what it can offer and shows the user.
4. User accepts or declines The user may share all, some, or none of the requested items.
5. Response maps back Artifacts cite fulfills[]; statuses cite each item id.
Core invariant: each request.items[].id must appear exactly once in response.requestStatus[]. Shared artifacts point back to one or more request item ids with fulfills[].

What makes a SMART request?

A request is a typed envelope plus an ordered list of request items. The envelope gives the conversation an identity and declares cross-cutting preferences. The items are the work the user sees.

Request envelope 
{
  "type": "smart-health-checkin-request",
  "version": "1",
  "id": "demo-all-of-the-above",
  "purpose": "Clinic check-in",
  "fhirVersions": ["4.0.1"],
  "items": [ ... ]
}
Envelope fields
type
Fixed discriminator for this payload family.
version
Payload schema version. Current value is "1".
id
Requester-chosen id echoed by response.requestId.
purpose
Human-readable reason to show in consent UI.
fhirVersions
FHIR versions the requester can consume.
items
The named checklist of clinical requests.

What makes a request item?

A request item is one consentable task. It has a stable id, a display label, a content description, and the response formats the requester accepts for that item.

Item shape 
{
  "id": "insurance",
  "title": "Insurance card",
  "summary": "Insurance card for billing",
  "required": true,
  "content": { ... },
  "accept": ["application/fhir+json"]
}
Item fields
id
Short stable name. Responses refer to this id.
title / summary
User-facing explanation of the request item.
required
Requester preference; the user still controls sharing.
content
The clinical thing requested.
accept
Media types the requester accepts for the artifact.

Specific FHIR profiles

Ask for resources conforming to known profiles, such as US Core Patient or C4DIC Coverage. 

{
  "id": "patient",
  "title": "Patient demographics",
  "content": {
    "kind": "fhir.resources",
    "profiles": [
      "http://hl7.org/fhir/us/core/StructureDefinition/us-core-patient"
    ]
  },
  "accept": ["application/fhir+json"]
}

Broad FHIR sharing

Omit filters when the requester is asking for any useful patient-specific FHIR resources the wallet and user can offer. 

{
  "id": "anything-useful",
  "title": "Relevant clinical history",
  "content": {
    "kind": "fhir.resources"
  },
  "accept": [
    "application/fhir+json",
    "application/smart-health-card"
  ]
}

Questionnaire

Ask the wallet to render a FHIR Questionnaire, accept user input, and return a QuestionnaireResponse. 

{
  "id": "intake",
  "title": "Migraine intake",
  "content": {
    "kind": "questionnaire",
    "questionnaire": {
      "resourceType": "Questionnaire",
      "title": "Migraine Check-in",
      "item": [
        {
          "linkId": "severity",
          "text": "Pain severity (0-10)",
          "type": "integer"
        }
      ]
    }
  },
  "accept": ["application/fhir+json"]
}
Canonical URLs with versions: the wallet may strip |version when fetching or checking family membership, but preserves the canonical as written in records, fixtures, returned FHIR resources, and logs.

What makes a SMART response?

A response is not just a pile of data. It is a typed envelope that links shared artifacts back to the original request and says what happened for every request item.

Response envelope 
{
  "type": "smart-health-checkin-response",
  "version": "1",
  "requestId": "demo-all-of-the-above",
  "artifacts": [ ... ],
  "requestStatus": [ ... ]
}
Envelope fields
type
Fixed discriminator for response payloads.
version
Response schema version. Current value is "1".
requestId
Must equal the original request id.
artifacts
Clinical payloads the user chose to share.
requestStatus
One outcome for every requested item.

Artifacts: the shared clinical content

An artifact is a concrete payload. It has its own id, a media type, and a fulfills[] array that maps it back to one or more request items.

FHIR JSON resource or bundle

{
  "id": "artifact-insurance",
  "mediaType": "application/fhir+json",
  "fhirVersion": "4.0.1",
  "fulfills": ["insurance"],
  "value": {
    "resourceType": "Coverage",
    "meta": {
      "profile": [
        "http://hl7.org/fhir/us/insurance-card/StructureDefinition/C4DIC-Coverage"
      ]
    }
  }
}

Raw FHIR artifacts carry fhirVersion outside the resource so the verifier can parse them consistently.

SMART Health Card

{
  "id": "artifact-ips-shc",
  "mediaType": "application/smart-health-card",
  "fulfills": ["ips"],
  "data": "shc:/567629..."
}

SMART Health Cards do not use outer fhirVersion; the verifier inspects the signed JWS payload.

QuestionnaireResponse

{
  "id": "artifact-intake",
  "mediaType": "application/fhir+json",
  "fhirVersion": "4.0.1",
  "fulfills": ["intake"],
  "value": {
    "resourceType": "QuestionnaireResponse",
    "questionnaire": "https://clinic.example/Questionnaire/migraine|2026-05",
    "status": "completed",
    "item": [
      {
        "linkId": "severity",
        "answer": [{ "valueInteger": 6 }]
      }
    ]
  }
}
Artifact field Meaning Important rule
id Responder-chosen artifact identifier. Unique inside the response.
mediaType How the verifier should parse the artifact. Must be one of the media types accepted by every fulfilled item.
fulfills Request item ids this artifact satisfies. Must be non-empty and reference real request item ids.
value Inline JSON content, such as FHIR JSON. Used for clear JSON artifacts.
data Compact encoded artifact, such as an SHC string. Useful for signed or binary-friendly payloads.
url Pointer to externally hosted content. Extension point; not required by the current demo.

Status: what happened for each item?

Statuses make partial sharing explicit. A verifier should not infer success from an artifact alone; it should check the status list too.

Status shape 
{
  "item": "insurance",
  "status": "fulfilled",
  "message": "Shared digital insurance card"
}
Allowed statuses
fulfilledThe item was satisfied.
partialSome, but not all, useful content was shared.
unavailableThe wallet has no matching data.
declinedThe user chose not to share it.
unsupportedThe wallet cannot handle that item shape.
errorThe wallet tried but failed.

A complete but readable example

This is the shape of the checked-in demo capture, simplified to show the application model without the raw mdoc bytes.

Requester sends

{
  "type": "smart-health-checkin-request",
  "version": "1",
  "id": "demo-all-of-the-above",
  "purpose": "Clinic check-in",
  "fhirVersions": ["4.0.1"],
  "items": [
    {
      "id": "patient",
      "title": "Patient demographics",
      "content": {
        "kind": "fhir.resources",
        "profiles": [".../us-core-patient"]
      },
      "accept": ["application/fhir+json"]
    },
    {
      "id": "insurance",
      "title": "Insurance card",
      "content": {
        "kind": "fhir.resources",
        "profiles": [".../C4DIC-Coverage"]
      },
      "accept": ["application/fhir+json"]
    },
    {
      "id": "ips",
      "title": "International Patient Summary",
      "content": {
        "kind": "fhir.resources",
        "profiles": [".../Bundle-uv-ips"]
      },
      "accept": [
        "application/smart-health-card",
        "application/fhir+json"
      ]
    },
    {
      "id": "intake",
      "title": "Intake form",
      "content": {
        "kind": "questionnaire",
        "questionnaire": { "resourceType": "Questionnaire", "title": "Migraine Check-in" }
      },
      "accept": ["application/fhir+json"]
    }
  ]
}

Responder returns

{
  "type": "smart-health-checkin-response",
  "version": "1",
  "requestId": "demo-all-of-the-above",
  "artifacts": [
    {
      "id": "artifact-patient",
      "mediaType": "application/fhir+json",
      "fhirVersion": "4.0.1",
      "fulfills": ["patient"],
      "value": { "resourceType": "Bundle", "entry": [ ... ] }
    },
    {
      "id": "artifact-insurance",
      "mediaType": "application/fhir+json",
      "fhirVersion": "4.0.1",
      "fulfills": ["insurance"],
      "value": { "resourceType": "Coverage" }
    },
    {
      "id": "artifact-ips-shc",
      "mediaType": "application/smart-health-card",
      "fulfills": ["ips"],
      "data": "shc:/..."
    },
    {
      "id": "artifact-intake",
      "mediaType": "application/fhir+json",
      "fhirVersion": "4.0.1",
      "fulfills": ["intake"],
      "value": { "resourceType": "QuestionnaireResponse" }
    }
  ],
  "requestStatus": [
    { "item": "patient", "status": "fulfilled" },
    { "item": "insurance", "status": "fulfilled" },
    { "item": "ips", "status": "fulfilled" },
    { "item": "intake", "status": "fulfilled" }
  ]
}

Where the wire details live

The SMART request and response above are embedded in a W3C Digital Credentials API exchange. This demo currently carries them through direct org-iso-mdoc, which adds CBOR, mdoc namespaces, HPKE encryption, SessionTranscript binding, MSO digests, and COSE signatures.
