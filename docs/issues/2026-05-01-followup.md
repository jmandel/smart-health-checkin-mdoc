## Follow-up items (closed)

### F1. `|version` not stripped in wallet kind classification

`wallet-android/.../SmartRequest.kt:98`

```kotlin
val profiles = stringList(content.optJSONArray("profiles"))
    .map { it.lowercase() }
    .toSet()
```

Lowercases each canonical but does not strip the `|version` suffix. So a
versioned canonical like
`http://hl7.org/fhir/us/insurance-card/StructureDefinition/C4DIC-Coverage|1.0.0`
fails the `endsWith("/structuredefinition/c4dic-coverage")` check and falls
through to `RequestKind.Unknown`.

Compare to `profilesFromCanonicals` on line 154, which already does
`.substringBefore('|').lowercase()` â€” the two paths diverge.

**Fix**: change the build-up at line 98 to
`.map { it.substringBefore('|').lowercase() }`.

**Severity**: bug, narrow trigger (no current fixture sends versioned
canonicals).

### F2. `validateResponseAgainstRequest` is exported but not auto-called

`rp-web/src/protocol/index.ts:1229`

The cross-validator is exported and exercised by tests, but consumers of
`openWalletResponse` get back a shape-valid response that has *not* been
checked against the original request. So a wallet that returns artifacts
referencing unknown ids, or omits `requestStatus` entries for required items,
will not be flagged unless the caller explicitly invokes the second function.

Two reasonable fixes:

- **Inline it**: have `openWalletResponse` accept the original request and
  run cross-validation by default. Most consumers already have the request
  in hand.
- **Make it harder to forget**: rename the consumer-facing entry point to
  something like `openAndValidateWalletResponse(req, â€¦)` that does both
  steps, and keep `validateResponseAgainstRequest` as the lower-level export.

**Severity**: API-shape concern, not a correctness bug.

### F3. `requiredString` does loose path â†’ key extraction

`wallet-android/.../SmartRequest.kt:204-208`

```kotlin
private fun requiredString(obj: JSONObject, path: String): String {
    val value = obj.opt(path.substringAfterLast('.'))
    require(value is String && value.isNotBlank()) { "$path missing or not a string" }
    return value
}
```

`path` is used both as the diagnostic string AND as the source of the actual
JSON key (via `substringAfterLast('.')`). Today it works because every call
site passes either a bare key (`"id"`) or `"items[i].title"`-style paths
where the trailing key is correct. A future caller that passes
`"items[0].content.kind"` would silently look up `"kind"` on the *outer*
object, not on the nested content object.

**Fix**: split the helper into `requiredString(obj, key, diagnosticPath)`
where the key and the path are explicit arguments.

**Severity**: cosmetic / robustness; not currently firing.

## Reference: canonical `|version` strip-vs-preserve rule

Came up while diagnosing F1. Worth adding to the spec or a short doc near the
protocol library so future contributors don't re-derive it.

| Operation | Action | Why |
| --------- | ------ | --- |
| HTTP fetch of a canonical | **Strip** | A canonical with `\|ver` is an identifier, not a URL. |
| IG / profile-collection membership (e.g. `profilesFrom`) | **Strip** | The IG question is version-agnostic by intent. |
| Wallet-side kind classification | **Strip** | Routing doesn't care about version (this is F1). |
| Profile de-dup / grouping | **Strip** | Two strings differing only by `\|ver` are the same logical profile for grouping. |
| `QuestionnaireResponse.questionnaire` | **Preserve** | Receiver needs to know the exact version that was answered. |
| `meta.profile` in returned resources | **Preserve** | Provenance â€” `X\|1.0.0` and `X\|2.0.0` are different claims. |
| Exact-match conformance checks | **Preserve consistently** | Compare both sides at the same level; never strip asymmetrically. |
| Test fixtures / logs / debug bundles | **Preserve** | Capture exactly what was sent. |

**Mental shorthand**:

- Going to the network or to "is this in IG X?" â†’ **strip**.
- Going into a record, a log, or back to the verifier â†’ **preserve**.
