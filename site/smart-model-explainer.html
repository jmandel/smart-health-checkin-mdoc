Requester and responder roles

The requester says what it wants for a check-in workflow. The responder decides what it can provide, asks the user, and returns the approved content.

Requester

A clinic, registration kiosk, patient portal, or relying-party web app. It owns the reason for the exchange, the requested item list, and the formats it can accept.

  • Creates a SMART Health Check-in request.
  • Invokes navigator.credentials.get().
  • Validates the response against the original request.

Responder

A patient wallet or health app. It owns available data, renders consent, and creates a SMART Health Check-in response.

  • Matches request items it understands.
  • Shows the user what will be shared.
  • Returns artifacts plus a status for every item.

The conversation is a checklist

This is not "give me one credential." It is "here is a checklist of consentable check-in tasks; tell me what the patient shared for each."

  1. Requester chooses a purpose. Example: clinic check-in, billing, visit intake, or prior authorization.
  2. Requester lists named items. Examples: patient demographics, insurance card, health summary, intake form.
  3. Responder matches the list. The wallet decides what it can offer and how each item should be shown.
  4. User accepts, edits, or declines. The user may share all, some, or none of the requested items.
  5. Responder maps everything back. Artifacts cite item ids in fulfills[]. Status rows cite item ids in requestStatus[].

Core invariant: each request.items[].id must appear exactly once in response.requestStatus[]. Artifacts point back to one or more item ids using fulfills[].

What makes a SMART request?

A request is a typed envelope plus an ordered list of items. The envelope identifies the conversation. The items are the work the user sees in the wallet. 

{
  "type": "smart-health-checkin-request",
  "version": "1",
  "id": "demo-us-core-checkin",
  "purpose": "Clinic check-in",
  "fhirVersions": ["4.0.1"],
  "items": [
    { "...": "request item" }
  ]
}
type
Fixed discriminator: smart-health-checkin-request.
version
Payload schema version. Current value is "1".
id
Requester-chosen id echoed by response.requestId.
purpose
Human-readable reason to show in consent UI.
fhirVersions
FHIR versions the requester can consume.
items
The named checklist of clinical requests.

What makes a request item?

A request item is one consentable task. It has a stable id, display text, a content selector, and a list of accepted response formats. 

{
  "id": "insurance",
  "title": "Insurance card",
  "summary": "Insurance card for billing",
  "required": true,
  "content": { "...": "what is being requested" },
  "accept": ["application/fhir+json"]
}
id
Short stable name. Responses refer to this id.
title / summary
User-facing display text for the wallet consent screen.
required
Requester preference. The user still controls sharing.
content
The clinical thing requested.
accept
Media types the requester accepts for the returned artifact.

Exact FHIR profile

Use profiles to ask for resources conforming to exact StructureDefinition canonicals, such as US Core Patient. 

{
  "id": "patient",
  "title": "Patient demographics",
  "content": {
    "kind": "fhir.resources",
    "profiles": [
      "http://hl7.org/fhir/us/core/StructureDefinition/us-core-patient"
    ]
  },
  "accept": ["application/fhir+json"]
}

Questionnaire

Use kind: "questionnaire" when the wallet should render a FHIR Questionnaire, accept user input, and return a QuestionnaireResponse. 

{
  "id": "intake",
  "title": "Migraine intake",
  "content": {
    "kind": "questionnaire",
    "questionnaire": {
      "resourceType": "Questionnaire",
      "title": "Migraine Check-in",
      "item": [
        {
          "linkId": "severity",
          "text": "Pain severity (0-10)",
          "type": "integer"
        }
      ]
    }
  },
  "accept": ["application/fhir+json"]
}

Profile families: asking for "any US Core profiles"

Exact profile requests use profiles. Broad implementation guide or profile-family requests use profilesFrom. Both are arrays, and when both are present they are additive: exact profiles highlight specific records of interest, but do not limit the broader profile-family request.

Any US Core resource the wallet can share 
{
  "id": "us-core-records",
  "title": "US Core records",
  "summary": "Patient records your app can share that conform to US Core profiles.",
  "content": {
    "kind": "fhir.resources",
    "profilesFrom": ["http://hl7.org/fhir/us/core"]
  },
  "accept": ["application/fhir+json"]
}
US Core family, narrowed by resource type 
{
  "id": "us-core-problems-meds-labs",
  "title": "Problems, medications, and labs",
  "content": {
    "kind": "fhir.resources",
    "profilesFrom": ["http://hl7.org/fhir/us/core"],
    "resourceTypes": ["Condition", "MedicationRequest", "Observation"]
  },
  "accept": ["application/fhir+json", "application/smart-health-card"]
}
profiles
Exact StructureDefinition canonicals, such as US Core Patient.
profilesFrom
Publication, implementation guide, or profile-collection canonical URLs, such as US Core as a family.
resourceTypes
Optional official FHIR resourceType narrowing within the profile family.
no selectors
Ask for any patient-specific FHIR resources the wallet can offer and the user chooses to share.

Version rule: strip |version when checking profile-family membership or fetching canonical content; preserve it in returned records, fixtures, logs, and exact profile checks.

What makes a SMART response?

A response is a typed envelope that links shared artifacts back to the original request and says what happened for every requested item. 

{
  "type": "smart-health-checkin-response",
  "version": "1",
  "requestId": "demo-us-core-checkin",
  "artifacts": [
    { "...": "shared clinical content" }
  ],
  "requestStatus": [
    { "...": "one outcome per item" }
  ]
}
requestId
Must equal the original request id.
artifacts
Clinical payloads the user chose to share.
requestStatus
One outcome for every requested item.

Artifacts are the shared clinical content

An artifact has its own id, a media type, and a fulfills[] array that maps it back to one or more request items.

FHIR JSON resource or bundle

{
  "id": "artifact-us-core",
  "mediaType": "application/fhir+json",
  "fhirVersion": "4.0.1",
  "fulfills": ["clinical-history"],
  "value": {
    "resourceType": "Bundle",
    "entry": [ ... ]
  }
}

SMART Health Card

{
  "id": "artifact-clinical-shc",
  "mediaType": "application/smart-health-card",
  "fulfills": ["clinical-history"],
  "value": {
    "verifiableCredential": ["shc:/567629..."]
  }
}

SMART Health Cards do not use outer fhirVersion; the verifier inspects the signed JWS payload.

Artifact field Meaning Important rule
id Responder-chosen artifact identifier. Unique inside the response.
mediaType How the verifier should parse the artifact. Must be accepted by every item in fulfills[].
fulfills Request item ids this artifact satisfies. Must be non-empty and reference real request item ids.
value Inline JSON content, such as FHIR JSON. Use for clear JSON artifacts.
data Compact encoded artifact, such as an SHC string. Use for signed or binary-friendly payloads.

Status says what happened for each item

A verifier should not infer success from artifacts alone. It should check the status list for every requested item. 

{
  "item": "insurance",
  "status": "fulfilled",
  "message": "Shared digital insurance card"
}
Status Meaning
fulfilledThe item was satisfied.
partialSome, but not all, useful content was shared.
unavailableThe wallet has no matching data.
declinedThe user chose not to share it.
unsupportedThe wallet cannot handle that item shape.
errorThe wallet tried but failed.

A complete example

This simplified example asks for any US Core records and an intake form, then returns a FHIR Bundle and QuestionnaireResponse. 

{
  "type": "smart-health-checkin-request",
  "version": "1",
  "id": "demo-us-core-and-intake",
  "purpose": "Clinic check-in",
  "fhirVersions": ["4.0.1"],
  "items": [
    {
      "id": "us-core-records",
      "title": "US Core records",
      "content": {
        "kind": "fhir.resources",
        "profilesFrom": ["http://hl7.org/fhir/us/core"]
      },
      "accept": ["application/fhir+json"]
    },
    {
      "id": "intake",
      "title": "Migraine intake",
      "content": {
        "kind": "questionnaire",
        "questionnaire": { "resourceType": "Questionnaire", "title": "Migraine Check-in" }
      },
      "accept": ["application/fhir+json"]
    }
  ]
}
{
  "type": "smart-health-checkin-response",
  "version": "1",
  "requestId": "demo-us-core-and-intake",
  "artifacts": [
    {
      "id": "artifact-us-core",
      "mediaType": "application/fhir+json",
      "fhirVersion": "4.0.1",
      "fulfills": ["us-core-records"],
      "value": { "resourceType": "Bundle", "entry": [ ... ] }
    },
    {
      "id": "artifact-intake",
      "mediaType": "application/fhir+json",
      "fhirVersion": "4.0.1",
      "fulfills": ["intake"],
      "value": { "resourceType": "QuestionnaireResponse" }
    }
  ],
  "requestStatus": [
    { "item": "us-core-records", "status": "fulfilled" },
    { "item": "intake", "status": "fulfilled" }
  ]
}
