SMART Health Check-in over W3C Digital Credentials API

Open prototype Spec: v1 draft MIT licensed

A web verifier — clinic, kiosk, or patient portal — asks a patient wallet for check-in information: FHIR resources, questionnaires, SMART Health Cards, and per-item status — in a user-approved, encrypted response.

Wallet response patient app

// SmartHealthCheckinResponse (excerpt)
{
  "type": "smart-health-checkin-response",
  "artifacts": [
    { "id": "a1",
      "type": "fhir_resource",
      "fulfills": ["insurance"],
      "data": { "resourceType":
        "Coverage", … } },
    { "id": "a2",
      "type": "questionnaire_response",
      "fulfills": ["intake"], … }
  ],
  "requestStatus": [
    { "id": "insurance", "status": "shared"   },
    { "id": "intake",    "status": "shared"   },
    { "id": "summary",   "status": "declined" }
  ]
}

Core invariant: each request.items[].id appears exactly once in response.requestStatus[]. Artifacts cite item ids in fulfills[], so a single artifact (a Bundle, a Health Card) can fulfill multiple items.

How it's wired

Three layers; both flows share the bottom two. The clinical content layer is transport-neutral. The browser transport is the W3C Digital Credentials API. The kiosk handoff is an extra wrapper for in-person check-in.

SMART request & response

Clinical content layer: items, artifacts, FHIR resources, questionnaires, SMART Health Cards, and per-item status. JSON, transport-neutral.

Model explainer → Draft spec →

Digital Credentials API

Browser transport for both flows. Wallet presentation uses direct org-iso-mdoc: HPKE encryption, COSE-signed mdoc, with the SMART payload carried inside.

Wire-protocol explainer → Byte fixtures →

Kiosk handoff wrapper

Cross-device only. A desktop QR points the phone to a signed, encrypted request and returns an encrypted SMART response back to the kiosk.
