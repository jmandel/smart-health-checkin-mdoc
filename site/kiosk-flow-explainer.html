Cross-device in-person check-in

Kiosk flow: same presentation, different handoff

The front-desk kiosk is not a second clinical protocol. It wraps the same SMART Health Check-in remote presentation flow so staff can start an in-person visit on a desktop while the patient presents from their phone.

Mental model

The same-device verifier already demonstrates the core presentation: a requester builds a SMART Health Check-in request, invokes navigator.credentials.get(), the patient approves in their wallet, and the requester opens and validates the response.

The kiosk demo keeps that exact phone-side presentation. The desktop kiosk only creates a short-lived request pointer and watches for an encrypted completion. The patient still uses their own phone and health app to present.

Same-device patient portal

One browser page owns the request, calls the Digital Credentials API, opens the encrypted response, and renders the received artifacts.

In-person kiosk wrapper

The desktop starts the request and displays a QR. The phone resolves the pointer, runs the same presentation, and returns an encrypted opened result for the desktop to render.

What happens in the kiosk demo

  1. Kiosk initiates a request. The creator page builds the full SMART request, signs it as compact ES256 JWS with a demo creator key, encrypts that JWS to the demo submission-service key, and stores the encrypted request through a provider.
  2. The QR carries only a pointer. The QR URL contains #r=<requestId>. It does not contain FHIR profile details, questionnaires, or response keys.
  3. The phone resolves and verifies. The submit page fetches the encrypted request by pointer, decrypts it locally, verifies the creator JWS against baked trusted demo public keys, and recovers the embedded SMART request.
  4. The phone runs the same presentation flow. The submit page reuses the React SmartCheckinButton, calls the Digital Credentials API, opens and validates the wallet response, then encrypts that opened result to the kiosk session key.
  5. The kiosk receives and reviews. The creator page watches provider submission rows, downloads each encrypted blob, decrypts it in browser memory, validates the route/session/request hash/nonce/form binding, and renders the same response review UI as the same-device requester page.

The provider abstraction

The static apps use a narrow transport interface so the kiosk flow is not hard-coded to Instant. The protocol code owns signing, encryption, request verification, response encryption, and binding checks. The provider only stores request rows, stores submission rows, serves encrypted blobs, and reports new submissions.

1. Create Kiosk signs/encrypts the request and stores an opaque request row.
2. Resolve Phone reads by pointer, decrypts, and verifies the creator signature.
3. Complete Phone encrypts the opened wallet result and writes a submission row.
4. Review Kiosk downloads, decrypts, validates bindings, and renders the result. 
type KioskTransportProvider = {
  name: string;
  appId: string;
  configured: boolean;
  writeRequest(...): Promise<KioskRequestRow>;
  readRequest(requestId): Promise<KioskRequestRow>;
  writeSubmission(...): Promise<KioskSubmissionRow>;
  downloadSubmissionBlob(row): Promise<Uint8Array>;
  useSubmissionRows(routeId): KioskSubmissionRows;
};

Current provider: InstantDB + Instant Storage

The checked-in demo provider uses InstantDB for small pointer rows and Instant Storage for encrypted response blobs. Instant is intentionally treated as untrusted transport.

What Instant can see: routing identifiers, timestamps, key IDs, content types, sizes, hashes, storage paths, and ciphertext.
What Instant should not see: the plaintext SMART request, requested FHIR profiles, questionnaires, opened wallet result, or returned clinical artifacts.

Demo public and private keys are checked into the repo so the static Pages app can run without a server. They demonstrate the architecture; they are not production secrets.
