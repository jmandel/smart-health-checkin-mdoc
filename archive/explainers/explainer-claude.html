0 Cast of characters & request/response data flow

Five entities collaborate to deliver one SMART Health Check-in exchange. Knowing who emits which bytes is the single biggest aid to reading the rest of this doc.

  • verifier rp-web — the relying-party browser app at rp-web/. Builds the request, opens the encrypted response, renders the FHIR data.
  • browser + host Chrome / Safari + Android Credential Manager — bridges navigator.credentials.get(...) from the page into the on-device picker, runs WASM matchers, dispatches a PendingIntent to the chosen wallet.
  • matcher shc-matcher.wasm — a 19 KB Rust WASM module at matcher/. Decides whether to surface our wallet entry by byte-searching the request.
  • wallet wallet-android — the holder app. Receives the request, drives the consent UI, builds the mdoc DeviceResponse, HPKE-seals it.
  • crypto cryptography — host crypto primitives invoked at multiple stages: ECDH P-256, SHA-256, HKDF, AES-128-GCM, ECDSA over P-256 (COSE_Sign1).

Request / response data flow

1. verifier builds DeviceRequest CBOR + EncryptionInfo CBOR; base64url-encodes both into navigator.credentials.get(...).
2. browser + host route the call to Android Credential Manager.
3. host invokes every registered WASM matcher of the matching type; the matcher byte-searches and emits one entry to the v2 picker surface.
4. User picks the SMART Health Check-in entry; host launches the wallet's HandlerActivity.
5. wallet CBOR-decodes the request, hydrates the consent UI from requestInfo.smart_health_checkin.
6. User picks what to share; wallet builds the SMART response JSON, packages it as the elementValue of a single mdoc IssuerSignedItem, signs the MSO and DeviceAuth, HPKE-seals the result against the verifier's key.
7. host hands the sealed bytes back to the page as a DigitalCredential.
8. verifier HPKE-opens, CBOR-decodes, walks down to the elementValue, parses the SMART response JSON, renders FHIR cards.

Reference protocol description: profiles/org-iso-mdoc.md. Reference implementations: rp-web/src/protocol/index.ts (TS), wallet-android/app/src/main/java/.../{DirectMdocRequest,SmartHealthMdocResponder,SmartMdocCrypto}.kt (wallet), matcher/src/lib.rs (matcher).

Per RFC 8152, COSE_Key labels are negative integers for curve-specific fields and positive integers for generic ones. kty=2 means EC2 (uncompressed (x, y) with implicit length per curve); crv=1 is P-256 with 32-byte coordinates. The x and y bstrs are exactly 32 bytes each (no leading 0x04 / no SEC1 prefix).

            
Verifier code that builds this: rp-web/src/protocol/index.ts → publicJwkToCoseKey (encodes a JWK as a 4-entry CBOR map). Wallet code that consumes it: wallet-android/.../SmartMdocCrypto.kt → publicKeyFromCose(cose) (rebuilds an ECPublicKey from the (x, y) bstrs to feed into HPKE).

            
Why no alg and no kid? The encryption purpose is implicit from the key being inside EncryptionInfo[1].recipientPublicKey. alg would constrain the wallet to a specific HPKE suite, but the suite is fixed by the protocol (DHKEM-P256 + HKDF-SHA256 + AES-128-GCM) so the field is redundant. The captured Safari-UA Mattr request confirms 4 fields exactly.

          

        

      

    
  


  
Total: 142 bytes. Notice the COSE_Key has only kty/crv/x/y — no alg, no kid, no use. That's confirmed by the captured Mattr Safari request and the wallet code accepts it as-is. Older docs that demanded a 6-field COSE_Key with alg=-25 (ECDH-ES) are wrong for this profile.

6 What the WASM matcher actually sees


  
host Credential Manager invokes every registered DC matcher of the matching type for every digital request. There is no host-side protocol pre-filter on the AndroidX registry-provider path — verified by decompiling RegistryManagerProviderPlayServicesImpl (passes protocolTypes = emptyList() to GMS).


  
So our matcher must self-filter. The bytes it gets via the credman import functions are roughly the JSON above, plus whatever the host wraps around it.


  
Concrete imports the matcher calls (matcher/src/bindings.rs):

  
    
    
  • credman::GetWasmVersion(&mut u32) — probe the host's ABI version. ≥ 2 means we can use the v2 set surface; 0/1 means v1 only.
    • 
    
  • credman::GetRequestSize(&mut u32) + GetRequestBuffer(*mut u8) — read the JSON body.
    • 
    
  • credman::GetCredentialsSize(...) + ReadCredentialsBuffer(...) — read the wallet's registered credentials blob (a small JSON we control).
    • 
  


  
What the matcher does with the request bytes (matcher/src/lib.rs::request_is_eligible):

  
    
    
  1. Substring-search the raw bytes for the literal "org-iso-mdoc". If absent, bail. (Defensive — the host doesn't pre-filter, but real DC API requests will have this string when the verifier asked for our protocol.)
    2. 
    
  2. Find the value of any "deviceRequest" JSON string field — both the plain shape ("deviceRequest":"...") and the legacy escaped-wrapper shape (\"deviceRequest\":\"...\") are accepted.
    3. 
    
  3. Base64url-decode that value into bytes.
    4. 
    
  4. Substring-search those bytes for the literal UTF-8 of the doctype "org.smarthealthit.checkin.1". CBOR text strings encode UTF-8 verbatim, so the doctype appears as a contiguous byte run inside the decoded CBOR.
    5. 
    
  5. If found, the matcher is eligible; otherwise zero entries.
    6. 
  


  
Why no real CBOR parser? Because we don't need one. The doctype is unique enough that any false-positive match is implausible, and a parser would balloon the WASM binary, pull in a hashing-default-hashmap dependency, and risk the WASI entropy trap that bit early SHL wallets.







  
7 What the matcher emits and why


  
matcher Eligibility check passes → the matcher emits one entry. How it emits depends on the host's WASM ABI version, probed via GetWasmVersion.


  
v2 (modern Credential Manager)

  
    
    
  1. credman_v2::AddEntrySet(set_id="smart-health-checkin-set", set_length=1) — declare a credential set.
    2. 
    
  2. credman_v2::AddEntryToSet(...) — title, subtitle, disclaimer, embedded 192×192 PNG icon (the SMART starburst), metadata JSON.
    3. 
    
  3. credman_v2::AddFieldToEntrySet(...) — one row of preview text shown in the picker ("Your health info: You choose what to share. Continue to review details in SMART Health Check-in.")
    4. 
  


  
v1 fallback (legacy hosts only)

  
    
    
  1. credman::AddStringIdEntry(cred_id, icon, ..., title, subtitle, disclaimer, NULL)
    2. 
    
  2. credman::AddFieldForStringIdEntry(cred_id, name, value)
    3. 
  


  
Why both paths? Because the modern AndroidX picker UI only reads from the v2 surface. A matcher that emits via v1 only is silently dropped on current devices — exactly the bug that produced "Your info wasn't found" until we added the version probe (see matcher/src/lib.rs::emit_one_entry).


  
Embedded icon

  
The icon argument is a raw PNG byte slice — matcher/assets/icon.png, a 192×192 RGBA rendering of the SMART starburst, generated by matcher/assets/build-icon.ts from the upstream smarthealthit.org SVG. The WASM module embeds it via include_bytes!("../assets/icon.png"); ~7.3 KB of the 19 KB matcher is just the icon.







  
8 The wallet handler's input from Credential Manager


  
wallet User taps the entry in the picker → host fires the PendingIntent we registered → HandlerActivity.onCreate runs in our app process. The activity calls PendingIntentHandler.retrieveProviderGetCredentialRequest(intent) and gets back a ProviderGetCredentialRequest.


  
Code: wallet-android/app/src/main/java/.../HandlerActivity.kt. The decoder/walker for the request is DirectMdocRequestParser in DirectMdocRequest.kt.


  
From the ProviderGetCredentialRequest the handler pulls:

  
    
    
  • callingAppInfo.packageName — Android-verified, e.g. com.android.chrome. Always trustworthy.
    • 
    
  • callingAppInfo.getOrigin(privAppsJson) — verified web origin only if the calling app's signing cert matches an entry in our allowlist. With "{}" it returns null for every caller, and we fall back to android-app:<packageName>.
    • 
    
  • req.selectedEntryId — the metadata JSON we emitted in AddEntryToSet, echoed back. Lets the wallet remember which credential the user picked when there are several.
    •

10 mdoc IssuerSignedItem and its tag-24 digest


  
wallet The SMART response JSON from §9 gets wrapped in an IssuerSignedItem per ISO 18013-5 §8.3.2.1.2.2, then tag-24-wrapped, then SHA-256-hashed. That hash goes into the MSO (next section).



IssuerSignedItem = {
  "digestID":         0,                     // index in MSO.valueDigests[ns]
  "random":           <bstr 16+ bytes>,    // unpredictable, per item
  "elementIdentifier": "smart_health_checkin_response",
  "elementValue":      "<the SMART response JSON tstr from §9>"
}


  
The matched static element name from §2 is echoed here: the verifier asked for smart_health_checkin_response in the request namespace, and the wallet replies with an item whose elementIdentifier is exactly that string. The verifier knows what to walk to.


  
tag-24 wrap

  
The encoded IssuerSignedItem CBOR is then wrapped in CBOR tag 24 (i.e. d8 18 followed by a bstr containing the IssuerSignedItem bytes). The reasons:

  
    
    
  • The MSO's valueDigests map stores SHA-256 over the tag-24-wrapped bytes, not the inner item alone.
    • 
    
  • The DeviceResponse later transports the same tag-24 wrap as the actual on-the-wire entry, so the verifier hashes the bytes it received and compares to the MSO's digest entry — no separate canonicalization step.
    • 
  


  
Code: wallet-android/app/src/main/java/.../SmartHealthMdocResponder.kt — see issuerSignedItemTag24Bytes = MdocCbor.tag24Bytes(MdocCbor.encode(issuerSignedItem)). The MSO digest is then SmartMdocCrypto.sha256(issuerSignedItemTag24Bytes).







  
11 The MobileSecurityObject and issuerAuth COSE_Sign1


  
The MobileSecurityObject (MSO) is the issuer-signed manifest that lets a verifier check that no element value has been tampered with.



MSO = {
  "version":         "1.0",
  "digestAlgorithm": "SHA-256",
  "docType":         "org.smarthealthit.checkin.1",
  "valueDigests": {
    "org.smarthealthit.checkin": {
      0: <sha-256 of the tag-24-wrapped IssuerSignedItem from §10>
    }
  },
  "deviceKeyInfo": {
    "deviceKey": <COSE_Key for the device-signed signature key>
  },
  "validityInfo": {
    "signed":     "2023-11-14T22:13:20Z",
    "validFrom":  "2023-11-14T22:13:20Z",
    "validUntil": "2023-11-15T22:13:20Z"
  }
}


  
The MSO is then tag-24-wrapped and signed by the issuer key as a COSE_Sign1:



issuerAuth = COSE_Sign1[
  protected:    bstr .cbor { 1: -7 },                  // ES256
  unprotected:  { 33: [<DER-encoded x509 cert chain>] },
  payload:      bstr .cbor (tag-24 of the MSO above),
  signature:    bstr ECDSA-P256-SHA256
]


  
For this debug harness the wallet generates an ephemeral issuer key pair per response and ships a self-signed cert in the x5chain. Verifiers report "issuer untrusted" but that's fine for demos.


  
Why two key pairs (issuer + device)? Per ISO 18013-5 the issuer key signs the static credential structure once at issuance time; the device key proves possession at presentation time and is bound by being included in deviceKeyInfo.deviceKey of the issuer-signed MSO. We collapse both into ephemeral wallet-side keys for the demo.


  
Code: SmartMdocCrypto.signCoseSign1(...) in SmartMdocCrypto.kt. The Sig_structure bytes signed are: cbor.encode([ "Signature1", protected_header_bstr, external_aad=h'', payload_bstr ]).

Why this layout?

  
    
    
  • The two leading nulls are mandated by ISO 18013-5's SessionTranscript schema. They're slots for offline-presentation engagement bytes that don't apply to the DC API.
    • 
    
  • The handover is ["dcapi", hash] — the Annex C "online" form. (Profile A's OpenID4VPDCAPIHandover form is byte-different and a wallet that mixes them up fails deviceSignature verification.)
    • 
    
  • Hashing handoverInfo commits to exactly the encryptionInfo bytes the wallet decoded — change a single nonce byte and the wallet's signature and the verifier's recomputation diverge.
    • 
    
  • origin is included indirectly via handoverInfo. The Profile B "dcapi" handover is sometimes described as not origin-binding — this is a slight simplification: we hash origin into the transcript, so the wallet's signature and the verifier's AAD do commit to a verifier-supplied origin string. (The wallet has to trust the host that origin belongs to the calling app, since without an allowlist it can't independently verify.)

13 The deviceSignature COSE_Sign1


  
The wallet's device key signs over a CBOR array containing the SessionTranscript bytes, the doctype, and an empty DeviceNameSpaces map. This is what proves "the holder of the device key authorized this presentation, in this session."



deviceAuthentication = cbor.encode([
  "DeviceAuthentication",
  <SessionTranscript bytes from §12>,
  "org.smarthealthit.checkin.1",                  // doctype
  cbor.tag24(cbor.encode({}))                     // empty DeviceNameSpaces
])

deviceAuthenticationBytes = cbor.tag24(deviceAuthentication)

deviceSignature = COSE_Sign1[
  protected:    bstr .cbor { 1: -7 },             // ES256
  unprotected:  { },                              // none
  payload:      deviceAuthenticationBytes,
  signature:    ECDSA-P256-SHA256(deviceKey, sig_structure)
]


  
Where sig_structure = cbor.encode([ "Signature1", protected_bstr, h'', deviceAuthenticationBytes ]) per RFC 9052.


  
This is the second COSE_Sign1 in the response (the first was issuerAuth in §11). The two are independent: issuerAuth proves the credential is intact, deviceSignature proves the holder agreed to present it now to this verifier.

15 HPKE-sealing the response


  
crypto The DeviceResponse plaintext is sealed with HPKE base mode (RFC 9180) under the verifier's recipient public key from §4.


  
Code: SmartMdocCrypto.hpkeSeal(...) in SmartMdocCrypto.kt. Hand-rolled (no JOSE / no HPKE library on Android) — about 100 lines of boilerplate over java.security primitives.


  
Suite

  
    
    
  • KEM: DHKEM(P-256, HKDF-SHA256) — kemId 0x0010
    • 
    
  • KDF: HKDF-SHA256 — kdfId 0x0001
    • 
    
  • AEAD: AES-128-GCM — aeadId 0x0001
    • 
    
  • info: SessionTranscript bytes from §12 (binds the seal to this exact session)
    • 
    
  • aad: empty
    • 
  


  
Algorithm

  
    
    
  1. Generate ephemeral wallet key pair (P-256). Public point in raw uncompressed form is the 65-byte enc.
    2. 
    
  2. ECDH(walletEphemeralPriv, recipientPub) → 32-byte dh.
    3. 
    
  3. HKDF-Extract / -Expand per RFC 9180 with KEM and HPKE labeled-extract/expand. Output: 16-byte AES key + 12-byte base nonce.
    4. 
    
  4. AES-128-GCM encrypt the DeviceResponse plaintext with that key + nonce + empty AAD. Output: ciphertext = body || 16-byte tag.
    5. 
  



HpkeSealResult = {
  enc:        <65 bytes — ephemeral wallet public point, 0x04 || X || Y>,
  cipherText: <DeviceResponse plaintext length + 16 byte tag>
}

17 The verifier opens the response


  
verifier The page that fired navigator.credentials.get awaits the promise and gets back a DigitalCredential object whose data is the JSON above. From there:


  
Code: rp-web/src/protocol/index.ts → openWalletResponse, inspectDcapiMdocResponse, inspectDeviceResponseBytes. The UI code in App.tsx walks the resulting structure into artifacts + answers and feeds each artifact's data to ResourceCard from app/ResourceCards.tsx.


  
    
    
  1. Verify credential.protocol === "org-iso-mdoc".
    2. 
    
  2. Base64url-decode data.response → CBOR bytes → walk: ["dcapi", { enc, cipherText }].
    3. 
    
  3. Recompute the SessionTranscript from the same (origin, encryptionInfo bytes) the page retained from request build. Same code path as §12 — buildDcapiSessionTranscript.
    4. 
    
  4. HPKE.Open(enc, cipherText, info=SessionTranscript, aad=empty) using the recipient private key the page retained from request build.
    5. 
    
  5. The decrypted plaintext is the DeviceResponse CBOR from §14. CBOR-decode → walk to the single document.
    6. 
    
  6. Validate the structural envelope:
      
      
        
    • document.docType matches "org.smarthealthit.checkin.1".
      • 
        
    • issuerAuth COSE_Sign1 verifies under the cert in x5chain. (Self-signed → issuer is "untrusted" but signature is intact — UI shows that distinction.)
      • 
        
    • Recompute SHA-256 over the tag-24 IssuerSignedItem bytes; compare against MSO.valueDigests[ns][0].
      • 
        
    • deviceSignature COSE_Sign1 verifies under MSO.deviceKeyInfo.deviceKey, signing over cbor.tag24(["DeviceAuthentication", SessionTranscript, docType, deviceNameSpacesBytes]).
      • 
      
    
    
    7. 
    
  7. Pull elementValue out of the lone IssuerSignedItem — that's the SMART response JSON.
    8. 
    
  8. For each artifact in smartResponse.artifacts, call <ResourceCard credentialId={artifact.id} resource={artifact.data} />. The renderer dispatches on resource.resourceType — Coverage gets a card-image grid (front/back from C4DIC SupportingImage extensions), Bundle gets a per-entry summary, QuestionnaireResponse gets a recursive Q&A walk, etc. Below the typed cards, the existing artifact-by-artifact JSON debug <details> blocks remain available for low-level inspection.
    9.

The two profiles share the same SMART JSON shapes but differ in protocol value, request envelope, SessionTranscript handover variant (OpenID4VPDCAPIHandover vs dcapi), and response envelope (compact JWE vs HPKE).
