Mental Model

The verifier asks for one mdoc credential. That mdoc credential is not a driver's license. It is our SMART Health Check-in document type. The request asks for one stable mdoc element, and the actual SMART request payload lives in ItemsRequest.requestInfo.smart_health_checkin as a JSON string.

1. RP page Builds navigator.credentials.get with an org-iso-mdoc request and an HPKE recipient key.
2. Browser Shows the trust prompt and tells Android the web origin, such as http://127.0.0.1:3000.
3. Credential Manager Runs wallet matchers and launches the selected wallet handler activity.
4. Wallet Decodes the request, shows consent, builds a self-attested mdoc DeviceResponse.
5. RP page HPKE-opens the response, validates mdoc hashes, and renders SMART artifacts.
Current successful signal: the verifier displays HPKE opened, digest matched, 4 artifacts, and 4 answered items. That means the browser origin, SessionTranscript, HPKE key schedule, and MSO digest all lined up for the current implementation.

Request Bytes

The active verifier request shape is direct Digital Credentials API mdoc:

{
  "requests": [{
    "protocol": "org-iso-mdoc",
    "data": {
      "deviceRequest": "<base64url CBOR DeviceRequest>",
      "encryptionInfo": "<base64url CBOR ['dcapi', {...}]>"
    }
  }]
}

SMART request JSON

The verifier starts with a plain JSON object: {"version":"1","items":[...]}. Each item is a FHIR profile or an inline/questionnaire URL request. This is user-facing clinical intent, not mdoc selective disclosure metadata.

ItemsRequest

The SMART JSON is stored as a CBOR text string at requestInfo.smart_health_checkin. The requested mdoc element is stable: nameSpaces.org.smarthealthit.checkin.smart_health_checkin_response = false. The false byte is the mdoc intentToRetain flag. 

{
  "docType": "org.smarthealthit.checkin.1",
  "nameSpaces": {
    "org.smarthealthit.checkin": {
      "smart_health_checkin_response": false
    }
  },
  "requestInfo": {
    "smart_health_checkin": "<SMART request JSON string>"
  }
}

DeviceRequest

The ItemsRequest is CBOR-encoded, wrapped in tag 24, and inserted into DeviceRequest.docRequests[0].itemsRequest. 

{
  "version": "1.0",
  "docRequests": [
    { "itemsRequest": Tag(24, bstr .cbor ItemsRequest) }
  ]
}

encryptionInfo

The verifier also sends a CBOR array with a nonce and recipient HPKE public key. The public key is a COSE EC2 P-256 key. 

[
  "dcapi",
  {
    "nonce": bstr,
    "recipientPublicKey": { 1: 2, -1: 1, -2: x, -3: y }
  }
]

Origin And SessionTranscript

AndroidX only reveals callingAppInfo.origin after the wallet passes a privileged caller allowlist to getOrigin(...). In this dev build, the wallet reflects the actual caller app/signature to prove the plumbing. A production build should use a maintained list of trusted browser package names and signing-cert fingerprints.

Important distinction: the privileged app allowlist is for browsers/apps that are allowed to assert web origins. It is not an allowlist of clinic origins. Once Chrome is trusted, any clinic origin still gets shown to the user and bound into the transcript.

Origin extraction

The wallet obtains http://127.0.0.1:3000 from callingAppInfo.getOrigin(privilegedAllowlistJson). If this falls back to android-app:com.android.chrome, the transcript will not match the verifier page.

dcapiInfo

Build an intermediate CBOR array from the exact base64url encryptionInfo string and the origin:

dcapiInfo = CBOR([
  encryptionInfoBase64Url,
  origin
])

Handover hash

Hash that encoded byte string:

handover = [
  "dcapi",
  SHA-256(dcapiInfo)
]

SessionTranscript

This is the exact CBOR byte string used later in DeviceAuthentication and HPKE:

SessionTranscript = CBOR([
  null,
  null,
  handover
])

DeviceResponse Bytes

The wallet wraps the SMART response JSON in one mdoc issuer-signed item, creates an MSO with a digest of that item, signs the MSO, signs a DeviceAuthentication payload bound to the SessionTranscript, and emits a CBOR DeviceResponse.

SMART response JSON

The wallet produces {version, artifacts, answers}. The artifacts list holds FHIR resources or other payloads. The answers map links each request item id to artifact ids.

IssuerSignedItem

The SMART response JSON is the elementValue for the one stable element. The item is canonical CBOR, then wrapped as tag 24. 

IssuerSignedItem = {
  "digestID": 0,
  "random": random16,
  "elementIdentifier": "smart_health_checkin_response",
  "elementValue": "<SMART response JSON string>"
}

issuerSignedItemTag24 = Tag(24, bstr .cbor IssuerSignedItem)

Value digest

The MSO digest is over the exact tag-24 bytes, not just the inner map:

valueDigest = SHA-256(issuerSignedItemTag24)

MobileSecurityObject

The MSO records the digest, docType, device public key, digest algorithm, and validity window. It is CBOR-encoded and then used as the payload of an issuer COSE_Sign1.

DeviceAuthentication

The device signature payload binds the response to the verifier's request transcript:

DeviceAuthentication = Tag(24, CBOR([
  "DeviceAuthentication",
  SessionTranscript,
  "org.smarthealthit.checkin.1",
  Tag(24, bstr .cbor {})
]))

DeviceResponse

DeviceResponse = {
  "version": "1.0",
  "documents": [{
    "docType": "org.smarthealthit.checkin.1",
    "issuerSigned": {
      "nameSpaces": {
        "org.smarthealthit.checkin": [issuerSignedItemTag24]
      },
      "issuerAuth": COSE_Sign1(MSO)
    },
    "deviceSigned": {
      "nameSpaces": Tag(24, bstr .cbor {}),
      "deviceAuth": { "deviceSignature": COSE_Sign1(DeviceAuthentication) }
    }
  }],
  "status": 0
}

HPKE And Credential Result

The wallet encrypts the plaintext DeviceResponse to the verifier public key from encryptionInfo.recipientPublicKey.

ParameterCurrent implementation
KEMDHKEM(P-256, HKDF-SHA256), suite id 0x0010
KDFHKDF-SHA256, suite id 0x0001
AEADAES-128-GCM, suite id 0x0001
HPKE infoSessionTranscript bytes
AES-GCM AADempty byte string in the current implementation
NonceHPKE base nonce for sequence number 0

The encrypted result is wrapped as another small CBOR array:

dcapiResponse = CBOR([
  "dcapi",
  {
    "enc": hpkeEphemeralPublicKeyRawP256,
    "cipherText": aesGcmCiphertextAndTag
  }
])

The wallet returns that through Credential Manager as:

{
  "protocol": "org-iso-mdoc",
  "data": {
    "response": "<base64url dcapiResponse CBOR>"
  }
}

Verification Checklist

Request

Decode device-request.cbor. Confirm docType, namespace, element id, intentToRetain=false, and SMART JSON in requestInfo.

Transcript

Recompute CBOR([encryptionInfoB64u, origin]), SHA-256 it, then compare session-transcript.cbor.

Response

Check value-digest.bin == SHA-256(issuer-signed-item-tag24.cbor) and that the MSO contains the same digest id and namespace.

Known Edges

AreaCurrent behaviorFuture hardening
Browser origin trust Dev build reflects the actual caller app/signature into the AndroidX privileged allowlist. Use an up-to-date allowlist of trusted browser package names and official signing cert fingerprints.
Issuer trust Self-signed issuer cert. Useful for integrity and debugging, not third-party attestation. Add real issuer/IACA/VICAL only if product requirements demand it.
HPKE binding SessionTranscript is HPKE info; current AES-GCM AAD is empty. Keep tests pinned to this behavior or deliberately migrate both RP and wallet if a target platform requires a different binding.
Request carrier requestInfo.smart_health_checkin is load-bearing. Keep the dynamic element fallback documented but disabled unless a platform hides requestInfo.
